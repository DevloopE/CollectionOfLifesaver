第 2 章 浮点数：计算机里的"实数"

上一章我让你接受了一句话："你写下 0.1 的瞬间，计算机存的不是 0.1，是它附近的一个特殊数。" 那时我没解释这个"附近"和"特殊"具体是什么。这一章打开盒子。

读这一章之前你会觉得"浮点数"是个有点神秘的工程话题。读完之后你会发现它其实是一件极朴素的事——把"科学计数法"搬到二进制下，加上一个全人类约定的存储格式。神秘感来自符号本身（11 位指数 / 52 位尾数 / 偏移量 1023），不是来自概念。我们会先从概念开始，再看符号。

2.1 0.1 在二进制下其实是无限循环小数

我们一起手算 0.1 的二进制。回忆十进制里怎么把 \(0.625\) 写成二进制：反复 ×2，看整数部分。

\(0.625 \times 2 = 1.25\) → 整数部分 1，余下 0.25。
\(0.25 \times 2 = 0.5\) → 整数部分 0，余下 0.5。
\(0.5 \times 2 = 1.0\) → 整数部分 1，余下 0。停。

从上往下读整数部分：\(0.625 = 0.101_{(2)}\)。漂亮、有限、精确。

现在试 0.1：

\(0.1 \times 2 = 0.2\) → 0，余 0.2
\(0.2 \times 2 = 0.4\) → 0，余 0.4
\(0.4 \times 2 = 0.8\) → 0，余 0.8
\(0.8 \times 2 = 1.6\) → 1，余 0.6
\(0.6 \times 2 = 1.2\) → 1，余 0.2
\(0.2 \times 2 = 0.4\) → 0，余 0.4 …… 等等，这不就回到第三行了吗？

对。从这里开始 0.2 → 0.4 → 0.8 → 0.6 → 0.2 这四步永远循环下去。所以

\[ 0.1_{(10)} = 0.0001100110011001100\ldots_{(2)} = 0.0\overline{0011}_{(2)}. \]

0.1 在二进制下是无限循环小数。就像 \(1/3 = 0.333\ldots\) 在十进制下是无限循环——并不是 1/3 不存在，而是它不能在十进制下用有限位表达。

你心里大概一惊：常用小数 0.1、0.2、0.3 在二进制下全部都是循环小数。在十进制下"漂亮"的数，到了二进制下漂亮不漂亮，是两码事。

计算机只有有限位。它必须在循环小数的某一位停下来。停下来意味着四舍五入。所以你以为存的是 0.1，其实存的是"截断到 53 位 mantissa 的二进制 0.0001100110011...001"——它和真 0.1 差大约 \(10^{-17}\)。

\[ \boxed{\ \text{你写下的小数 } \neq \text{ 计算机里存的小数}.\ } \]

这是上一章那个 0.30000000000000004 的全部根源。两个被四舍五入过的数加起来，"错了一点点" 是必然的。

2.2 浮点 = 二进制的"科学计数法"

你看过科学计数法 \(1.5 \times 10^3 = 1500\)。它由三块构成：符号（+ 或 −）、尾数（1.5）、指数（3）。这种格式有一个好处——同一种存储能表达从极小到极大的数。\(10^{-300}\) 和 \(10^{+300}\) 都是 "尾数 × 10 的某次方"，存储方式一样。

把这套搬到二进制。一个浮点数是

\[ (-1)^s \times m \times 2^e, \]

其中 \(s\) 是符号位（0 或 1）、\(m\) 是尾数（一个介于 1 和 2 之间的二进制数）、\(e\) 是指数（一个整数）。比如 1.5 用浮点表达是 \((-1)^0 \times 1.1_{(2)} \times 2^0\)：尾数 1.1（二进制），指数 0。

"尾数永远落在 [1, 2) 之间" 这一条叫规格化（normalized）。它有两个好处：第一，每个数表达方式唯一（不然 \(1.5 \times 2^0 = 0.75 \times 2^1 = 0.375 \times 2^2\)，含糊不清）。第二，最高位永远是 1，这一位不用存——既然它必然是 1，存它就是浪费。这条小聪明叫隐含位（implicit leading bit）。

所以一个浮点数的实际存储就三块字段：

  • 符号位 \(s\)：1 个 bit。
  • 指数 \(e\)：若干 bit，存的是偏移后的无符号整数（避免存负数）。
  • 尾数小数部分 \(f\)：若干 bit，对应 \(m = 1.f_1 f_2 f_3 \ldots f_k\) 中那一串 \(f_i\)。

到这一步还没碰任何"标准"——这是任何"二进制科学计数法"必须有的字段。下面那一节讲为什么全人类后来同意了一份具体规格。

2.3 IEEE 754：全人类的约定

1985 年之前，每家做 CPU 的公司各搞一套浮点格式。Intel、IBM、CDC、Cray、DEC——精度不同、舍入规则不同、特殊值处理不同。同一段程序在不同机器上算出不同结果。这件事让所有写跨平台数值代码的人头疼。

1985 年，电气电子工程师学会通过了 IEEE 754 标准。从那以后所有主流 CPU（包括你现在用的）都遵守它。这个标准的精髓只两条：

  • 规定字段宽度——单精度 32 位（1 + 8 + 23），双精度 64 位（1 + 11 + 52）。
  • 规定舍入规则——默认"向最近的可表达数舍入，平手时取偶数尾数"（round-half-to-even）。

双精度（double）是绝大多数科学计算的默认。我们专门看双精度的字段：

\[ \underbrace{s}_{1\,\text{位}} \quad \underbrace{e_1 e_2 \ldots e_{11}}_{11\,\text{位指数}} \quad \underbrace{f_1 f_2 \ldots f_{52}}_{52\,\text{位尾数}} \]

指数字段存的是 \(e + 1023\)（偏移量 1023，让 \(e\) 可以为正可以为负但字段本身永远非负）。所以指数字段从 1 到 2046 表示 \(e\) 从 \(-1022\) 到 \(+1023\)；字段 0 与 2047 留作特殊用途（下面 2.5 节）。

尾数 52 位 + 隐含 1 位 = 总共有 53 位有效精度。换算成十进制大约 \(\log_{10}(2^{53}) \approx 15.95\) 位。这就是为什么人们说 double 有"约 16 位精度"。

用一个例子把规则跑实。\(1.5 = 1.1_{(2)} \times 2^0\)。
符号位 \(s = 0\)。
指数 \(e = 0\)，存为 \(0 + 1023 = 1023 = 01111111111_{(2)}\)。
尾数 \(m = 1.1\)，去掉隐含的 1，剩下 \(.1\)，补零到 52 位：
1000000000000000000000000000000000000000000000000000。

合在一起 64 位：

0 01111111111 1000000000000000000000000000000000000000000000000000

这就是 1.5 在内存里的精确样子。

0.1 是什么样？指数和尾数都不"漂亮"——你回到 2.1 节那个无限循环。截断到 52 位之后存进去，再加上隐含位，复原回来不再是 0.1，是 \(0.1 + \delta\)，\(\delta \approx 5.55 \times 10^{-18}\)。这个 \(\delta\) 就是上一章 0.30000000000000004 末尾那个 \(4 \times 10^{-17}\) 的近因。

2.4 机器精度 \(\varepsilon_{\text{mach}}\)

"双精度有 16 位精度" 这句话需要被精确化。我们定义

\[ \boxed{\ \varepsilon_{\text{mach}} := \min \{\varepsilon > 0 : \text{fl}(1 + \varepsilon) > 1\},\ } \]

其中 \(\text{fl}(x)\) 表示 "把 \(x\) 舍入到最近浮点"。用人话：\(\varepsilon_{\text{mach}}\) 是使得 \(1 + \varepsilon\) 在浮点下还不等于 1 的最小 \(\varepsilon\)。

对 double，相邻于 1 的下一个浮点数比 1 大 \(2^{-52} \approx 2.22 \times 10^{-16}\)（指数为 0，最低位尾数变 1）。所以 \(\varepsilon_{\text{mach}} = 2^{-52}\)。

你可以五行代码验：

eps = 1.0
while 1.0 + eps / 2 != 1.0:
    eps /= 2
print(eps) # 2.220446049250313e-16

请你真跑一遍。看到那个数自己浮出来，比记一个常数有教育意义。

\(\varepsilon_{\text{mach}}\) 的几个常被搞混的事：

  • 它不是最小的可表达正数。最小的正规化浮点是 \(2^{-1022} \approx 10^{-308}\)，比 \(\varepsilon_{\text{mach}}\) 小得多。\(\varepsilon_{\text{mach}}\) 是相对精度——不是"小到何"，而是"相邻浮点之间的相对间距"。
  • 它依赖于"在 1 附近"测量。在 \(2^{100}\) 附近，相邻浮点之间的绝对间距大得多（\(2^{100} \cdot \varepsilon_{\text{mach}} \approx 10^{14}\)）；但相对间距永远是 \(\varepsilon_{\text{mach}}\)。这是浮点最重要的几何性质：相对精度均匀，绝对精度按数量级伸缩。
  • 每一次 \(+, -, \times, /\) 在 IEEE 754 下的舍入误差都不超过 \(\varepsilon_{\text{mach}} \cdot |\text{真结果}|\)。这是上一章我说"每个 op 把答案舍到最近浮点"的精确版本。

整个数值分析的"误差分析"语言后面会一直围绕 \(\varepsilon_{\text{mach}}\) 转。下一章你会看到形如 \(\|\text{算出来} - \text{真答案}\| \le C \cdot \kappa \cdot \varepsilon_{\text{mach}} \cdot \|\text{真答案}\|\) 的不等式——把"算法做了多少 op"凑成 \(C\)，把"问题先天难度"凑成 \(\kappa\)（条件数），\(\varepsilon_{\text{mach}}\) 是无可逃避的物理常数。

2.5 \(0\)、\(\infty\)、NaN、subnormal

IEEE 754 的指数字段留了两端给特殊值。这一节快速过一遍，不展开。

零。指数字段 0、尾数 0、符号 0 = +0；同样符号 1 = −0。两个零在比较下相等（+0 == -0），但保留符号是为了区分"从正方向趋零"和"从负方向趋零"——在算 \(1/x\) 的极限时这区别决定你得 \(+\infty\) 还是 \(-\infty\)。

无穷大。指数字段全 1（即 2047）、尾数 0：表示 \(\pm \infty\)。来自上溢（一个数太大装不下）或除以 0。\(\infty\) 参与运算遵循直觉：\(\infty + 1 = \infty\)、\(\infty \cdot 0 = \text{NaN}\)、\(1/\infty = 0\)。

NaN（Not a Number）。指数字段全 1、尾数非 0：表示"算了一个没意义的东西"，比如 \(0/0\)、\(\infty - \infty\)、\(\sqrt{-1}\)、\(\log(-1)\)。NaN 的脾气特别——它不等于任何东西，包括它自己。NaN == NaN 返回 False。这条小怪癖恰好提供了一个检测 NaN 的方法：if x != x: ...。

Subnormal（次正规数）。指数字段 0、尾数非 0：表示比最小正规化数（\(2^{-1022}\)）还小的数。这些数的最高位不再是隐含的 1，所以精度递减。它们存在是为了"渐进下溢"——当数趋零时，精度逐渐损失而不是突然变 0。这是 IEEE 754 的小巧妙设计。

这四种特殊值你只需在 debug 时知道它们存在。"算到 NaN" 通常意味着流水线某处 0/0 或 \(\infty - \infty\)，往回找。

2.6 三条"看起来等价但其实不"的浮点等式

纯数学里你以为天经地义的事，在浮点下不一定成立。三个最重要的：

(F1) 加法不结合。 \((a + b) + c\) 在浮点下不一定等于 \(a + (b + c)\)。具体：

\(a = 10^{16}, b = -10^{16}, c = 1\)。
\((a + b) + c = 0 + 1 = 1\)。
\(a + (b + c) = 10^{16} + (-10^{16} + 1)\)。但 \(-10^{16} + 1\) 在双精度下就等于 \(-10^{16}\)（1 太小，落在 \(10^{16}\) 的精度之外）。所以 \(a + (b + c) = 10^{16} + (-10^{16}) = 0\)。

\[ \boxed{\ (a + b) + c \neq a + (b + c) \ \text{在浮点下普遍成立}.\ } \]

这条让"什么顺序求和"变成一个有正确选择的问题。求一个长正负序列的和，先把同号的加在一起、再相加，比从左到右一路加准。这就是后面会用到的 Kahan 求和算法背后的一种朴素直觉。

(F2) 乘法分配律有时也踩坑。 \(a \cdot (b + c)\) 与 \(a \cdot b + a \cdot c\) 一般也不严格相等——尽管偏差通常比加法不结合小一些。

(F3) "等于" 不是布尔判别。前面我们已经看到 NaN != NaN。更普遍地，永远不要用 == 比较两个浮点是否相等。代之以

\[ |a - b| < \varepsilon \cdot \max(|a|, |b|, 1) \]

之类的容差比较。\(\varepsilon\) 取多大依问题——通常取 \(10^{-10}\) 到 \(10^{-12}\) 之间，给操作误差留点空间又不至于把"真不等"的情形漏掉。

这三条（不结合 / 不分配 / 不能 ==）是浮点新手最容易踩的雷。你把它们写在便签上贴显示器边，一年内会感谢自己。

2.7 一个让你以后受用的小习惯

读完这一章，你应该养成三个反射动作：

  • 不用 == 比较浮点。用 abs(a - b) < tol。
  • 不把数量级差很大的数混在一起加（除非用 Kahan 这类补偿算法）。
  • 看见有"两个几乎相等的数相减"的代码，停下来想这里会不会灾难性抵消（上一章 1.2 节）。如果会，找个代数等价的形式重排。

这三个习惯能拦掉 80% 的数值 bug。剩下 20% 是更微妙的——比如长链算法的误差累积、迭代法的收敛诊断、矩阵几乎奇异——本书后面 22 章每章都在讲这些。

下一章我们要把"每次操作的小误差" 怎么传到"整段计算的大误差" 量化。这就是误差分析。我们会区分两种立场——向前（"我算出来的答案离真答案多远？"）和向后（"我算出来的答案，是哪一道扰动了的题的真答案？"）。这种区分是数值分析最深刻的视角之一，比"算法稳不稳"还要根本。
